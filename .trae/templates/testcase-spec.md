# 测试用例 Markdown 规范

本规范定义测试点（`testpoints/`）和测试用例（`testcases/`）的 Markdown 格式。配套 XMind 导出见 `scripts/md2xmind.py`。

## 核心风格：画布 2（按业务需求/迭代组织）

**默认风格**。每个用例都挂在一个明确的**业务需求**下，结构：

```
Sheet 标题（如"运营平台"）
└── 产品/系统名（如"小二当家"）
    └── 业务需求集合（如"迭代用例"）
        └── 迭代版本/日期（如"20260730"）
            └── 【模块名】业务需求描述      ← H3 业务需求
                └── 功能模块               ← H4 涉及模块
                    └── 子功能              ← H5 功能点
                        └── 验证场景        ← H6 用例标题
                            └── 预期结果    ← H6+ 列表
```

**业务需求标题规范**：`【模块名】业务名 + 关键行为描述`
- 示例：`【跟进记录】政务：外呼录音AI预生成 - 任务&预生成`
- 示例：`【小二当家】政务，企业一户式展示多个定位地址`

> 用户指定其他风格时（如"按功能模块全量沉淀"→ 画布 1 风格），可切换。但**默认按业务需求/迭代组织**。

## 测试点文档（`testpoints/`）

文件名：`{需求名称}-测试点-{序号}.md`

```markdown
# 测试点分析文档

## 文档信息
- 需求名称 / 需求版本 / 分析日期 / 分析人员 / 文档来源

## 需求概述
## 功能模块分析
### [模块名称]模块
#### [功能点名称]功能
- 功能描述 / 业务场景 / 涉及角色
##### 功能验证点 / 边界验证点 / 异常验证点 / 集成验证点 / 兼容性验证点 / 性能验证点
- TP_[模块]_[功能]_[序号]: [测试点名称]
  - 验证要点 / 优先级 / 关联需求 / 前置条件 / 测试数据 / 备注

## 测试点统计
## 模块统计
```

测试点 ID 格式：`TP_[模块缩写]_[功能缩写]_[序号]`，例如 `TP_REF_001`。

## 测试用例文档（`testcases/`）

文件名：`{需求名称}-测试用例-{序号}.md`

### 6 级 Markdown 标题层级（XMind 兼容）

```
# [业务/产品集合]                H1 - Sheet 标题（系统/产品名）
## [迭代/产品]                   H2 - 迭代/产品名
### 【模块名】业务需求描述        H3 - 业务需求（"【】"开头）
#### [功能模块]                  H4 - 涉及模块
##### [子功能/功能点]             H5 - 功能点
###### 预期结果                  H6 - 预期结果
- 预期结果详情1（无序列表项）
```

> **注意**：画布 2 风格下 H3 是业务需求节点，**不**直接挂预期结果。预期结果挂在 H5 功能点下的 H6。

### 强制元数据块（H5 功能点下必填）

每个 H5 后**必须**有元数据块（XMind 节点 notes + marker 用）：

```markdown
##### PC端，最近一次跟进及侧边跟进记录列表组件
> 优先级: 高 | 关联需求: REQ_20260730_001 | 前置条件: 业务方传入id列表 | 测试数据: id=[1001,1002,1003]
###### 预期结果
- 列表展示对应id的跟进记录
- 按时间倒序排列
- 抽屉组件正常打开
```

**元数据字段**（全部必填，不填视为不完整）：

| 字段 | 取值 | XMind 映射 |
|---|---|---|
| 优先级 | **高/中/低**（必填） | 🔴/🟡/🟢 marker |
| 关联需求 | `REQ_XXX_NNN` 或禅道 story ID | 节点 notes |
| 前置条件 | 自由文本（无则填"无"） | 节点 notes |
| 测试数据 | 自由文本（无则填"无"） | 节点 notes |

格式严格：`> 优先级: 高 | 关联需求: ... | 前置条件: ... | 测试数据: ...`（用 ` | ` 分隔）

## 模块拆分原则

**核心规则**：根据**功能复杂度和测试独立性**决定拆分粒度。

### 简单需求（少拆，深度 4-5 层）

适用：单一功能、1-2 个角色、不超过 5 个验证点的小需求。

```
H1 业务集合
└── H2 业务需求
    └── H3 【模块】业务名
        └── H4 功能模块（= 业务模块）
            └── H5 用例
                └── H6 预期
```

**不要**为了"对齐"而强行拆出多个功能模块。

### 复杂需求（按需拆，深度 5-7 层）

适用：多功能集成、多角色、多页面的需求。

拆分依据（满足任一就拆）：
- 涉及**多个功能模块**（每个模块独立测试）
- 涉及**多个角色**（管理员/用户/游客等权限差异）
- 涉及**多个平台**（PC 端/移动端/H5，需分开测试）
- 涉及**多个 API 接口**（每个接口独立验证）

```
H1
└── H2
    └── H3 【模块】业务名
        ├── H4 模块 A → H5 功能点 → H6 用例
        ├── H4 模块 B → H5 功能点 → H6 用例
        └── H4 模块 C → H5 功能点 → H6 用例
```

### 反模式（避免）

- ❌ 每个 H4 下只有 1-2 个用例（拆得太碎）
- ❌ 5 层以下就出现"验证点"中间层（应该合并到 H5 标题里）
- ❌ 同一功能在多个 H4 下重复出现（应合并）
- ❌ 业务需求名过长（应该精炼，或用简称）

## 用例设计方法

1. **等价类划分** — 有效/无效等价类各取代表
2. **边界值分析** — 最小/最大/临界值/空值
3. **判定表** — 多条件组合
4. **场景法** — 真实用户操作流程
5. **错误猜测** — 经验补充遗漏
6. **状态迁移** — 明确状态转换的系统

## 验证点类型 → 用例策略

| 测试点类型 | 生成的用例类型 |
|---|---|
| 功能验证点 | 正向用例（功能正常） |
| 边界验证点 | 边界用例（最小/最大/临界） |
| 异常验证点 | 异常用例（错误输入/系统异常） |
| 集成验证点 | 集成用例（模块/接口交互） |
| 兼容性验证点 | 兼容性用例 |
| 性能验证点 | 性能用例 |

**一对多映射**：一个测试点通常展开为多个用例场景。

## 质量检查清单（生成后必检）

- [ ] 业务需求标题以"【模块名】"开头
- [ ] 每个 H5 功能点下都有完整的元数据块（4 个字段都填）
- [ ] 模块拆分粒度合理（不过度拆分、不过度合并）
- [ ] 6 级标题层级严格正确
- [ ] 用例场景独立、粒度适中
- [ ] 预期结果具体可验证（避免"应该/可能"）
- [ ] 关键测试数据体现在 H5 标题或元数据中
