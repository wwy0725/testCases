# 测试点分析规则 (Test Point Analysis Rules)

## 概述 (Overview)
本规则文件用于指导如何从需求内容和开发设计中识别和提取测试点。测试点是需要验证的功能要点，不包含具体的测试步骤和数据。

## 整体流程 (Overall Process)

### 第一阶段：文档解析与内容提取
使用 Playwright MCP 工具解析输入文档，提取完整的需求和设计信息。

#### 1.1 文档类型识别
根据输入文档的类型，采用不同的解析策略：

**需求文档 (PRD)**
- 识别需求背景、目标、功能描述
- 提取业务规则、约束条件
- 识别用户故事、验收标准

**开发设计文档**
- 识别技术实现方案
- 提取接口定义、数据结构
- 识别业务逻辑、状态流转

**功能现有逻辑文档**
- 识别现有功能逻辑
- 提取历史实现细节
- 识别兼容性要求

#### 1.2 Playwright MCP 解析流程
```markdown
**步骤1**: 使用 playwright_navigate 导航到文档URL
**步骤2**: 使用 playwright_run_code 执行页面内容提取
**步骤3**: 提取页面完整文本内容，包括：
  - 标题和章节结构
  - 表格数据
  - 列表项
  - 代码示例
  - 图表说明
**步骤4**: 对提取的内容进行结构化整理
```

#### 1.3 特殊内容解析方法
需求文档中经常包含图片、流程图、代码、伪代码、跳转链接等特殊内容，这些内容往往包含了关键的测试信息，需要逐一解读分析。

##### 1.3.1 图片内容解析
**识别方法**
- 通过页面快照识别图片元素
- 提取图片的 alt 文本或描述
- 识别图片周围的文字说明

**解析策略**
- **UI界面图**: 分析界面布局、按钮位置、字段排列
- **原型图**: 提取交互流程、页面跳转关系
- **示例图**: 识别示例数据、典型使用场景
- **架构图**: 分析系统架构、模块关系、数据流向

**测试点提取**
- 从界面图提取UI元素验证点
- 从原型图提取交互流程验证点
- 从示例图提取数据格式验证点
- 从架构图提取集成验证点

##### 1.3.2 流程图解析
**识别方法**
- 识别流程图元素（开始/结束节点、处理节点、判断节点、连接线）
- 提取流程图中的文字说明
- 分析流程图的分支和循环

**解析策略**
- **业务流程图**: 分析业务流程、状态转换、异常处理
- **数据流程图**: 分析数据流向、数据转换、数据存储
- **系统流程图**: 分析系统交互、接口调用、消息传递

**测试点提取**
- 从业务流程图提取状态转换验证点
- 从数据流程图提取数据传递验证点
- 从系统流程图提取集成验证点
- 从分支条件提取边界验证点
- 从异常处理提取异常验证点

##### 1.3.3 代码和伪代码解析
**识别方法**
- 识别代码块（通常在 `<code>`、`<pre>` 标签中）
- 提取代码注释和说明
- 识别伪代码的关键逻辑

**解析策略**
- **接口定义**: 提取接口参数、返回值、错误码
- **业务逻辑**: 分析条件判断、循环、异常处理
- **数据结构**: 提取字段定义、数据类型、约束条件
- **伪代码**: 分析业务规则、算法逻辑、边界条件

**测试点提取**
- 从接口定义提取参数验证点
- 从业务逻辑提取功能验证点
- 从数据结构提取数据验证点
- 从条件判断提取边界验证点
- 从异常处理提取异常验证点

##### 1.3.4 跳转链接解析
**识别方法**
- 识别文档中的超链接
- 提取链接的文本描述
- 分析链接的上下文

**解析策略**
- **相关需求链接**: 识别关联的需求文档，提取相关测试点
- **技术文档链接**: 识别技术实现文档，提取技术细节
- **示例链接**: 识别示例页面，提取典型场景
- **API文档链接**: 识别接口文档，提取接口定义

**测试点提取**
- 从相关需求提取功能验证点
- 从技术文档提取集成验证点
- 从示例提取场景验证点
- 从API文档提取接口验证点

##### 1.3.5 表格数据解析
**识别方法**
- 识别表格元素（`<table>`、`<tr>`、`<td>`）
- 提取表格的表头和行数据
- 分析表格的结构和关系

**解析策略**
- **参数表格**: 提取参数定义、类型、约束、说明
- **状态表格**: 提取状态定义、转换条件、处理逻辑
- **配置表格**: 提取配置项、默认值、取值范围
- **数据表格**: 提取示例数据、字段说明、关系

**测试点提取**
- 从参数表格提取参数验证点
- 从状态表格提取状态转换验证点
- 从配置表格提取配置验证点
- 从数据表格提取数据验证点

##### 1.3.6 列表项解析
**识别方法**
- 识别列表元素（`<ul>`、`<ol>`、`<li>`）
- 提取列表项的文本内容
- 分析列表的层级结构

**解析策略**
- **功能列表**: 提取功能描述、功能关系
- **规则列表**: 提取业务规则、约束条件
- **场景列表**: 提取使用场景、操作步骤
- **要求列表**: 提取验收标准、质量要求

**测试点提取**
- 从功能列表提取功能验证点
- 从规则列表提取规则验证点
- 从场景列表提取场景验证点
- 从要求列表提取验收验证点

#### 1.4 多文档整合策略
当提供多个文档时，按以下优先级整合信息：
1. **需求文档** - 提供业务需求和功能描述
2. **开发设计文档** - 提供技术实现细节
3. **功能现有逻辑文档** - 提供现有功能上下文

**整合原则**：
- 需求文档为主，设计文档为辅
- 设计文档补充技术细节和边界条件
- 现有逻辑文档提供兼容性和回归测试要点
- 处理文档间的冲突时，以最新版本的需求文档为准
- 整合特殊内容时，优先保留详细描述的版本

### 第二阶段：需求拆分与功能点识别
将整合后的需求内容按照业务逻辑拆分为功能模块和功能点。

#### 2.1 功能模块划分原则
**按业务领域划分**
- 用户管理模块
- 订单处理模块
- 支付结算模块

**按用户角色划分**
- 管理员功能
- 普通用户功能
- 第三方集成功能

**按系统层次划分**
- 前端交互功能
- 后端接口功能
- 数据处理功能

#### 2.2 功能点识别规则
**识别核心业务动作**
- CRUD操作：创建、读取、更新、删除
- 业务动作：登录、注册、支付、审核、发布
- 状态变更：启用/禁用、上线/下线、通过/拒绝

**识别业务规则**
- 条件判断："当满足X条件时，执行Y操作"
- 约束限制："用户名长度不超过20个字符"
- 权限控制："只有管理员可以删除数据"

**识别数据验证**
- 数据格式验证：邮箱格式、手机号格式
- 数据完整性：必填字段、关联数据
- 数据一致性：跨模块数据同步

#### 2.3 功能点组织结构
```markdown
### [功能模块名称]模块

#### [功能点1]功能
- 功能描述：[简要描述该功能点的业务目的]
- 业务场景：[描述该功能点的典型使用场景]
- 涉及角色：[列出涉及的用户角色]

#### [功能点2]功能
- 功能描述：[简要描述该功能点的业务目的]
- 业务场景：[描述该功能点的典型使用场景]
- 涉及角色：[列出涉及的用户角色]
```

### 第三阶段：测试点提取与分类
在每个功能点下，按照验证类型提取测试点。

#### 3.1 测试点分类体系

**功能验证点 (Functional Verification Points)**
- 核心业务功能验证
- 数据处理逻辑验证
- 业务规则验证
- 状态变更确认

**边界验证点 (Boundary Verification Points)**
- 输入参数边界
- 数据量限制
- 时间范围约束
- 权限边界

**异常验证点 (Exception Verification Points)**
- 错误输入处理
- 系统异常响应
- 网络异常处理
- 资源不足处理

**集成验证点 (Integration Verification Points)**
- 模块间交互验证
- 外部系统集成验证
- 数据传递验证
- 接口调用验证

**兼容性验证点 (Compatibility Verification Points)**
- 向后兼容性
- 多版本兼容
- 数据迁移兼容
- 接口兼容

**性能验证点 (Performance Verification Points)**
- 响应时间
- 并发处理
- 资源占用
- 数据量处理

#### 3.2 测试点提取规则

**规则1：功能动作识别**
从需求中识别核心功能动作，每个动作对应一个测试点：
- **CRUD操作**: 创建、读取、更新、删除
- **业务动作**: 登录、注册、支付、审核、发布等
- **状态变更**: 启用/禁用、上线/下线、通过/拒绝等

**规则2：业务规则提取**
将需求中的业务规则转化为验证点：
- **条件判断**: "当满足X条件时，执行Y操作"
- **约束限制**: "用户名长度不超过20个字符"
- **权限控制**: "只有管理员可以删除数据"

**规则3：数据验证点识别**
识别需要验证的数据相关要点：
- **数据格式验证**: 邮箱格式、手机号格式
- **数据完整性**: 必填字段、关联数据
- **数据一致性**: 跨模块数据同步

**规则4：异常场景覆盖**
从需求中推导可能的异常情况：
- **输入异常**: 空值、超长、特殊字符
- **系统异常**: 网络中断、服务不可用
- **权限异常**: 无权限访问、会话过期

**规则5：边界条件识别**
识别需要验证的边界条件：
- **数值边界**: 最小值、最大值、临界值
- **长度边界**: 最小长度、最大长度
- **时间边界**: 开始时间、结束时间
- **数量边界**: 最小数量、最大数量

**规则6：集成场景识别**
识别需要验证的集成场景：
- **模块间交互**: 模块A调用模块B
- **外部系统集成**: 第三方API调用
- **数据传递**: 数据在不同模块间传递
- **接口调用**: 接口参数、返回值验证

#### 3.3 测试点详细描述模板
```markdown
**测试点ID**: TP_[模块]_[功能]_[序号]
**测试点名称**: [简洁描述需要验证的功能点]
**测试分类**: [功能验证/边界验证/异常验证/集成验证/兼容性验证/性能验证]
**优先级**: [高/中/低]
**验证要点**: [具体需要验证什么]
**关联需求**: [对应的需求条目]
**前置条件**: [执行该测试点需要满足的条件]
**测试数据**: [测试需要的数据要求]
**备注**: [其他需要说明的信息]
```

## 测试点定义 (Test Point Definition)

**测试点**: 从需求中提取的需要验证的功能要点或验证项，描述"需要测试什么"而不是"如何测试"。

### 测试点 vs 测试用例
- **测试点**: 用户登录功能
- **测试用例**: 输入正确用户名密码 → 点击登录 → 验证跳转到主页

## 测试点提取示例 (Test Point Extraction Example)

### 需求示例
```
用户注册功能需求：
1. 用户可以通过邮箱和手机号注册账户
2. 用户名长度为3-20个字符，不能包含特殊字符
3. 密码长度为8-16位，必须包含字母和数字
4. 邮箱格式必须正确
5. 手机号必须为11位数字
6. 注册成功后发送验证邮件
```

## 使用指南 (Usage Guidelines)

1. **文档解析**: 使用 Playwright MCP 解析输入文档，提取完整内容
2. **需求整合**: 整合多个文档的信息，形成完整的需求视图
3. **功能点划分**: 按业务逻辑将需求拆分为独立的功能点
4. **测试点提取**: 在每个功能点下按验证类型生成测试点
5. **结构组织**: 按照输出格式规范组织测试点结构
6. **优先级排序**: 根据业务重要性设置优先级
7. **质量检查**: 使用质量检查清单验证测试点质量
8. **评审确认**: 与需求方确认测试点覆盖完整性
9. **持续维护**: 需求变更时及时更新测试点

## 注意事项 (Notes)

### 文档解析注意事项
- 确保提取完整的文档内容，包括表格、列表、代码等
- 注意文档的版本信息，确保使用最新版本
- 多文档整合时，注意信息的一致性和冲突处理
- 保留文档来源信息，便于追溯

### 测试点提取注意事项
- 测试点关注"测试什么"而非"如何测试"
- 保持测试点的原子性，一个点验证一个要素
- 确保测试点与需求的可追溯性
- 测试点应为后续测试用例设计提供清晰指导
- 统计数据应定期更新，确保与实际测试点保持一致

### 质量保证注意事项
- 定期进行测试点评审，确保质量
- 建立测试点版本管理机制
- 保持测试点与需求文档的同步更新
- 建立测试点变更记录
